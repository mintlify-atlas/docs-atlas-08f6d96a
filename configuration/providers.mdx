AI Balancer supports four LLM providers out of the box: Grok (xAI), Cerebras, Google Gemini, and OpenRouter. At startup, the gateway runs auto-discovery — it scans all registered provider adapters, validates each API key, and adds only the providers that authenticate successfully to the active pool. Providers that fail validation are logged and skipped; the rest are immediately available for Round Robin load balancing.

<Note>
At least one provider must be configured with a valid API key before the gateway can serve chat requests. If no providers register successfully, the server starts but returns `503 Service Unavailable` on all chat endpoints.
</Note>

## How auto-discovery works

You don't need to explicitly register providers in code. The `core/discovery.py` module automatically finds every service adapter in the `services/` directory and attempts to validate its API key during the startup lifecycle. The sequence is:

1. All service adapters are instantiated.
2. Each adapter calls its `validate_key()` method, which makes a lightweight authenticated request to the provider's API.
3. Adapters that return a valid response are added to the `ServiceRegistry`.
4. Adapters with invalid keys (HTTP 401/403) are logged as errors and excluded.
5. The list of active providers is logged: `Proveedores registrados: [...]`.

You can verify which providers are active at any time by calling `GET /v1/providers`.

```bash
curl http://localhost:8000/v1/providers \
  -H "X-API-Key: YOUR_API_KEY"
```

  <Tab title="Grok (xAI)">
    Grok is xAI's language model, accessed through the OpenAI-compatible API at `https://api.x.ai/v1`.

    **Environment variables**

    | Variable | Default |
    |---|---|
    | `GROK_API_KEY` | — (required to activate) |
    | `GROK_MODEL_ID` | `grok-2` |
    | `GROK_BASE_URL` | `https://api.x.ai/v1` |

    **Minimal `.env` configuration**

    ```bash
    GROK_API_KEY=xai-your-key-here
    # GROK_MODEL_ID=grok-2  # optional, this is the default
    ```

    | Failure threshold | 3 failures |
    | Recovery timeout | 30 seconds |

## Enabling and disabling specific providers

By default, every provider with a valid API key is registered. Use `ENABLED_PROVIDERS` to restrict which providers are active without removing their keys from `.env`:

```bash
# Enable only Grok and OpenRouter
ENABLED_PROVIDERS=grok,openrouter

# Enable all providers with valid keys (default behavior)
ENABLED_PROVIDERS=
```

Provider names are lowercase and match the values shown in `GET /v1/providers`: `grok`, `cerebras`, `gemini`, `openrouter`.

## What happens when a key is invalid

If a provider's API key fails validation at startup, the gateway:

1. Logs an error with the HTTP status code and reason: `API key inválida para grok: NO REGISTRADO - 401 Unauthorized`.
2. Skips that provider — it is not added to the registry.
3. Does not prevent other providers from registering.
4. Continues serving requests using the remaining valid providers.

To fix a skipped provider, update the key in `.env` and restart the gateway. You can check the current active provider list without restarting by calling `GET /v1/providers`.
